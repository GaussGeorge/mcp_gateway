### 评审决定：Minor Revision / Weak Accept (小修 / 弱接收)

### 1. 文章概述 (Summary)
[cite_start]本文针对多步大语言模型（LLM）智能体在利用模型上下文协议（MCP）调用外部工具时面临的过载控制问题，提出了一种名为 PlanGate 的网关系统 [cite: 4, 5][cite_start]。现有的请求级限流机制会导致“级联计算浪费”（cascading compute waste），即会话中途被拒绝会导致前期消耗的计算资源全部作废 [cite: 4, 27][cite_start]。PlanGate 通过提供会话感知的准入控制来解决这一问题，核心机制包括：针对 Plan-and-Solve 智能体的原子化预检与预算锁定，针对 ReAct 智能体的沉没成本动态定价（基于二次衰减函数计算折扣），以及防御恶意节点的声誉机制 [cite: 6, 7, 8]。

### 2. 优势与创新点 (Strengths)
* [cite_start]**问题切入点精准**：文章敏锐地捕捉到了从传统微服务（无状态 RPC）向 LLM 智能体架构（多步依赖会话）演进过程中产生的核心痛点 [cite: 110, 111][cite_start]。对“级联计算浪费”的定义和量化具有较高的实际工程指导意义 [cite: 109]。
* [cite_start]**系统设计完备且自洽**：针对现阶段最主流的两种智能体规划模式（Plan-and-Solve 与 ReAct），分别设计了静态前置拦截和动态折扣补偿机制，能够在不侵入现有 MCP 服务端的前提下，作为透明反向代理独立部署 [cite: 34, 103, 104]。
* [cite_start]**实验评估全面详实**：实验设计不仅包含了大量 Mock 数据下的压力测试与消融实验（如重负载比例、并发敏感度），还横跨了 GLM-4-Flash 和 DeepSeek-V3 两个具有不同吞吐特性的真实 LLM 平台进行验证 [cite: 8, 345, 374][cite_start]。尤其是在 Exp9 中展示了系统在高达 1000 并发下的极高鲁棒性（仅产生 0.4 次级联失败）[cite: 364]。

### 3. 不足与改进建议 (Weaknesses & Suggestions for Improvement)
尽管工程实现非常扎实，但本文在部分理论深度和系统可扩展性论证上存在不足，建议在最终修改版中予以解决：

* **缺乏分布式扩展性（Scalability）的论证**：
    [cite_start]目前的评估均基于运行在单一节点（WSL2）上的 Go 网关实例 [cite: 268][cite_start]。在生产环境中，API 网关通常是多节点集群部署。由于 PlanGate 依赖本地的 `sync.Map` 来维护会话预留状态和动态价格表 [cite: 628, 629][cite_start]，一旦扩展为多节点，状态同步带来的延迟可能会严重影响动态定价的及时性和准入的一致性。建议作者增加一小节，探讨分布式部署下的状态一致性开销或引入 Redis 等外部存储的性能理论影响 [cite: 629, 630]。
* **定价与衰减函数的理论论证薄弱**：
    [cite_start]文章在 Exp8 中通过经验性的消融实验确定了二次衰减函数（$1/(1+K^2\alpha)$）的效果最优 [cite: 336, 622][cite_start]。然而，作为一篇高水平学术论文，仅凭经验结论不够严谨。作者也在“Limitations”中承认缺乏理论证明 [cite: 624]。建议引入博弈论或机制设计（Mechanism Design）的视角，简要论证该定价曲线如何满足用户的激励相容性（Incentive Compatibility）。
* **部分超参数缺乏自动化调节能力**：
    [cite_start]系统依赖多个硬编码或手动调参的变量，例如沉没成本系数 $\alpha$、价格更新步长、以及声誉系统的容忍阈值（$\tau_{ban}=0.3$）[cite: 243, 640][cite_start]。在真实且流量分布高度动态的场景中，手动微调是不现实的。建议作者探讨基于在线反馈机制自动调整 $\alpha$ 值的可能性，以彻底消除手动配置负担 [cite: 641]。
* **声誉机制（Reputation System）应对复杂攻击的局限性**：
    [cite_start]安全拦截主要依赖 EWMA 追踪客户端声明的 `X-Agent-ID` [cite: 235, 236][cite_start]。如果攻击者实施女巫攻击（Sybil Attack）或频繁轮换伪造的 Header 身份，当前的声誉机制将形同虚设 [cite: 626][cite_start]。建议在 3.7 节或讨论部分明确声明，该功能需与上游的 API Key 强鉴权网关结合使用才能保证真正的安全防范 [cite: 626]。
