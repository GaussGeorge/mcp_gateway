# 🏆 MCP 服务治理系统：全链路实验落地 Master Plan (V6 究极版)

> **核心方法论**：双模式评测 (无菌/战场) + 强一致性异步发压 + 细粒度物理与状态捕获 + 容灾自动化流水线 + 大样本统计保障。

## 📍 Phase 1: 基础设施与环境基线（排雷与组网期）
**目标：排除所有非代码因素（网络、依赖、端口）的干扰，建立极其稳定的测试床（Testbed），并确保异常处理机制 100% 健壮。**

* **[ ] 1.1 依赖与配置代码化 (Infrastructure as Code)**
    * **动作**：编写 `env_setup.sh`，一键安装 Python 3.10+, Go 1.21+, 及 vLLM/Ollama 等依赖。将所有网关 IP、端口、核心权重参数提取到 `config.yaml`。
    * **意义**：保证实验环境跨机器迁移时 100% 一致。
* **[ ] 1.2 组建高可用 P2P 物理内网**
    * **动作**：放弃公网直连或不稳定的 frp，使用 **Tailscale** 或 **ZeroTier** 将云服务器（网关+发压机）与本地 A4000 算力节点组建成虚拟局域网。
    * **验证**：执行 10 分钟连续 Ping 与轻量 HTTP 压测，确保 RTT 延迟极度平稳（如 15ms±2ms）且无丢包。
* **[ ] 1.3 核心逻辑的“无菌”单点与 💡[新增] 错误码专项验证**
    * **动作**：仅使用 `calculate`, `web_fetch`(Mock), `mock_heavy`。
    * **正常测试**：手动下发 JSON-RPC，确认 NG、SRL、DP 的计费和路由逻辑正确。
    * **异常测试（错误码拦截）**：
        1. 下发非法参数（如 `budget=-1`, 缺少 `tool_name`），验证返回 **400 Bad Request**。
        2. 下发极端超时请求（故意让 `mock_heavy` sleep 60s），验证返回 **504 Gateway Timeout**。
    * **意义**：防止在几万并发压测时，因为网关未能正确捕获异常导致协程崩溃或数据记录错位。跑通后打标 `git tag -a func_v1.0`。

## 📍 Phase 2: 学术级评测引擎构建（核心武器库研发）
**目标：打造一个不卡死、全追踪、带物理监控，且能精细区分“熔断”与“崩溃”的工业级发压机。**

* **[ ] 2.1 物理监控与 OOM 告警探针 (`monitor.py`)**
    * **开发**：使用 `psutil` 记录 CPU/内存 RSS；使用 `py3nvml` 记录 A4000 的显存。
    * **告警线**：设定显存占用 90% 触发告警，立即写入日志 `[CRITICAL] GPU OOM Risk triggered at QPS=xxx`。这将在论文中作为 DP 策略拯救系统的“物理铁证”。
* **[ ] 2.2 强一致性全异步发压机 (`load_generator.py`)**
    * **协程级随机一致性**：实验启动瞬间，基于全局 Seed 预生成包含 10 万个请求详细参数的数组，消费者协程仅按时间发包，消除调度随机性。
    * **元数据追踪**：生成 `meta.json`（记录 Git Hash、Seed、网关类型），确保 3 个月后实验仍可完美复现。
* **[ ] 2.3 💡[新增] 精细化状态记录与明确超时规则**
    * **开发**：在请求发起侧设置绝对的 HTTP Timeout 阈值（如 15s）。在 CSV 日志中不仅记录 `is_rejected`，还**新增一列 `is_timeout` (Boolean)**。
    * **意义（论文决胜点）**：这让你的数据能讲出极其高级的故事：
        * `is_rejected = True` (HTTP 429)：代表网关**主动健康保护**（好指标，DP 网关的功劳）。
        * `is_timeout = True` (HTTP 504 / Client Timeout)：代表网关保护失败，后端**阻塞崩溃**（坏指标，NG/SRL 的原罪）。

## 📍 Phase 3: 本地“无菌实验室”容灾自动化跑批（爆兵与采数期）
**目标：无人值守完成近百组变量正交实验，产出完美平滑的学术曲线。**

* **[ ] 3.1 锁定“无菌模式”配置**
    * 仅开启 `STERILE` 模式：固定 80% `calculate` + 20% `web_fetch`(Mock)，重量工具仅限可控的 `mock_heavy`。
* **[ ] 3.2 终极自动化编排脚本 (`run_experiments.py`)**
    * **开发**：用 `itertools.product` 定义参数网格。引入 `tqdm` 进度条实时显示预计剩余时间。
    * **💡[新增] 实验顺序防灾优化**：按照 **“先短后长，先核心后边缘”** 的顺序排列网格。例如：先跑每组仅 2 分钟的 Exp1（负载模式对比），再跑每组需要 5 分钟的 Exp2（Heavy Ratio 敏感性），最后跑参数扫描 Exp7。
    * **意义**：如果停电或宿主机死机，你已经把论文最核心的几张图的数据稳稳攥在手里了。
* **[ ] 3.3 无人值守执行与标定**
    * 具备断点续跑和清理钩子（清理僵尸进程）。跑完收集所有 CSV，打标 `git tag -a sterile_eval_done`。

## 📍 Phase 4: 云端“真实战场”大样本攻防（决战期）
**目标：引入混沌流量，触发系统崩溃边缘，抓取极端的 P999 尾延迟。**

* **[ ] 4.1 切换至“真实战场”配置**
    * 开启 `BATTLEFIELD` 模式：挂载真实的 `llm_reason`、`doc_embedding` 和 `python_sandbox`。采用 Trace-driven 权重分布。
* **[ ] 4.2 10万级大样本脉冲压测 (捍卫 P999 的尊严)**
    * **动作**：注入远超 A4000 承受极限的脉冲并发（Step/Burst）。单次压测请求总量**必须超过 10 万次**，为 P99.9 尾延迟提供坚实的统计学支撑。
* **[ ] 4.3 捕获生死对比数据**
    * **NG 策略**：捕获超高的 `is_timeout=True` 比例，以及 `monitor.py` 疯狂弹出的 `[CRITICAL] GPU OOM Risk`。
    * **DP 策略**：捕获大量的 `is_rejected=True`，同时验证 GPU 活在安全水位，且成功放行的请求 P999 保持在极低位。
    * 打标：`git tag -a cloud_eval_done`。

## 📍 Phase 5: 严谨统计学汇总与出图（成果固化期）
**目标：一键生成 14 张符合 CCF-B 级顶会审美的 PDF 矢量图，形成无懈可击的证据链。**

* **[ ] 5.1 规范化绘图脚本 (`plot_figures.py`)**
    * 统一配色（DP=#2196F3, SRL=#FF9800, NG=#F44336）和字体（Helvetica）。全量导出 `.pdf` 矢量图。
* **[ ] 5.2 严格的统计显著性处理**
    * 计算 Goodput 和延迟均值时，强制添加 **均值 ± 标准差 (SD)** 的误差棒。使用 ANOVA/t-test 计算 P 值并在图表自动标星（*p<0.05）。
* **[ ] 5.3 恪守“异常值”红线原则**
    * **绝对禁止**：在计算 P95、P99、P999 或绘制 Exp8（真实战场）图表时，严禁使用任何 3σ 过滤。必须把拥塞产生的真实长尾暴露出来。
    * **受限使用**：仅在无菌模式平滑纯 CPU 处理的中位数（P50）时可谨慎使用 3σ 过滤，且必须在论文中书面声明。
