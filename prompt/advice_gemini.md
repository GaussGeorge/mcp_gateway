### 一、 必须增加的图表（针对 CCF-B/C 审稿人的口味）

#### 1. 增加图9：消融实验对比图（Ablation Study）
* **为什么需要**：在你的设计中，DP（Dynamic Pricing）策略包含了两个非常核心的创新机制：**Regime Detector（自适应档位切换）** 和 **Tool Weights（工具权重）**。审稿人一定会问：“你的效果好，到底是因为基础的动态定价，还是因为加了这几个高级机制？”
* **图表设计**：建议做一个柱状图或雷达图。对比四个变体：
    * DP-Full（完整版）
    * DP - NoRegime（固定使用单一参数档位，不随负载模式切换）
    * DP - NoWeights（所有 MCP 工具权重相等）
    * SRL（静态基线）
* **核心故事**：证明你的每一个创新模块（尤其是为 MCP 定制的权重机制）都在为提升整体 Goodput 做出不可替代的贡献。

#### 2. 增加图10：网关自身的系统开销图（Gateway Overhead）
* **为什么需要**：你目前在 “B. 辅助资源指标” 中监控了**后端（Backend）**的 CPU 和内存。但这不够！系统论文中，引入一个智能网关最忌讳的就是“网关自身成为性能瓶颈”。DP 需要计算 P-controller、维持衰减窗口、做 Regime 检测，这些都会消耗网关所在机器的 CPU。
* **图表设计**：CDF 图或表格。展示请求在**纯网关层**的处理延迟（Gateway Processing Latency，通常是微秒 $\mu s$ 级别），以及在极高并发下网关进程的 CPU 占用率。
* **核心故事**：证明你的 Go 语言网关实现极其轻量，复杂的算法并没有带来显著的计算开销。

#### 3. 增加图11：绝对并发扩展性图（Scalability Test）
* **为什么需要**：你现在的 Exp2 测试了 `heavy_ratio` 的敏感性，这代表了“流量成分”的变化。但系统评测还需要看“绝对并发量”的变化。当并发客户端数（Concurrency Clients）从 10 激增到 1000 时，系统会怎样？
* **图表设计**：X轴为并发连接数/发送速率（RPS），Y轴为吞吐量。这是一个标准的“折点图（Hockey Stick Chart）”。
* **核心故事**：证明随着绝对压力上升，NG 会在早期崩溃，SRL 会过早触顶平直，而 DP 能够把整个系统的吞吐量推向理论上的物理极限。

### 二、 现有图表的优化建议

1.  **图4 和 图8（时间序列与动态价格图）的视觉强化**：
    * 你的 `Regime Detector` 有三个状态：`Bursty`, `Periodic`, `Steady`。建议在时间序列图（图4 和 图8）的**背景上使用不同的透明底色（Shading）**来标识当前网关处于哪个状态。
    * **效果**：这能极具说服力地向审稿人展示：当外界负载波形（如 Sine 波）发生变化时，你的网关是如何敏锐地“察觉”并自动切换底层算法参数的。
2.  **明确 Goodput（有效吞吐量）的绝对 C 位**：
    * 在传统的 API 网关论文里，看重的是 Throughput (RPS)。但在 MCP 服务治理中，拦截掉 1 个消耗 5000ms 的重量级请求，放行 50 个消耗 100ms 的轻量级请求，RPS 升高了，但实际系统做的“有效功”可能是一样的。
    * 因此，你在 Exp3 中设计的 `Goodput = Σ(通过请求 × budget 权重)` 是整篇论文立论的核心。建议在 **图1（摘要表）** 和 **图5（敏感性分析）** 中，把 Goodput 作为最粗、最显眼的线条。

### 三、 下一步该开始写什么组件？（回答文档末尾的问题）

既然你的 Go 核心代码（3 个网关策略、7 个工具）都已经就绪，整个实验成败的瓶颈就在于**如何产生完全符合你期望的流量**。

强烈建议你**优先实现：Load Generator（负载生成器）**。

1.  **原因**：它是一切数据的源头。你需要用 Python (基于 `asyncio` 和 `aiohttp`) 编写一个高并发发压器。
2.  **核心难点**：
    * 它必须能够精确控制 `heavy_ratio` 和 `high_budget_ratio`。
    * 它必须能按照数学模型（Step 的阶跃函数、Sine 的正弦函数、Poisson 的指数分布时间间隔）精确控制发包的速率（Arrival Rate）。
    * 它不能因为后端阻塞而自己卡死（必须是完全异步非阻塞的，否则你测出的延迟其实是你发压机自己排队的延迟）。

只要 Load Generator 跑通了，结合你现有的 Go 网关，你马上就可以在本地看到终端输出的日志，直观地感受到 Regime 的切换和 Price 的跳动了。
